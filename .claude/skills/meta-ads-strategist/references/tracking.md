## 1. Иерархия метрик

| Уровень | Метрики | Источник истины | Роль |
|---|---|---|---|
| 1. North star | **Pipeline value** по рынку/месяцу | CRM | Месячная оценка, куда реально идут деньги |
| 2. Целевые | **CPQL** (cost per qualified lead), **cost per meeting**, lead→qualified % | CRM + spend (join по `lead_id`) | Основа kill/scale между рынками и адами |
| 3. Тактическая | **CPL** (строго First Conversion) | Ads Manager / Insights API | In-flight сигнал эффективности, не цель |
| 4. Диагностические | CPM, link CTR, hook rate, frequency, FTIR, form open/completion rate | Ads Manager / Delivery Insights | Объясняют, ПОЧЕМУ двигаются уровни выше. Никогда не таргеты |

Механика, почему так:

- **Lead-objective bidding без downstream-сигнала карабкается к самому дешёвому form-fill** — Audience Network, pre-fill accidental submits. Низкий CPL в instant-form кампаниях сам по себе подозрителен. Лид за €50, который квалифицируется, лучше лида за €5, который нет; время сейлзов — узкое место (project-context.md).
- **Meta видит максимум 7 дней после клика.** Наш цикл — месяцы: на длинных циклах 20–40% конверсий выпадает за окно атрибуции. Значит qualified/meetings/deals считаются ТОЛЬКО в CRM (join по `meta_lead_id`), а цифры Ads Manager — сигнал доставки и эффективности, не бизнес-правда.
- **Пост-Andromeda качество сигнала само влияет на доставку** — чистый CAPI-фидбек снижает cost per quality lead на 15–21% (данные Meta). Иерархия метрик — это ещё и то, чем мы кормим оптимизатор (§5).
- CPL смотрим только как **First Conversion**: All Conversions завышает лиды на ~20–25% повторными сабмитами — CPL выглядит лучше реальности.

### Сигнал vs шум на наших объёмах

500–3000 EUR/мес/рынок → десятки лидов и единицы qualified в месяц на рынок. Дисциплина окон:

| Метрика | Минимальное окно для решения | Почему |
|---|---|---|
| CPL | rolling 3–5 дней; дни 1–2 после запуска/правки — «expensive noise» | однодневные качели — самая частая причина преждевременных kill |
| CPM / CTR / completion | WoW против **собственного trailing-30d baseline**, алерт при отклонении >25–30% | абсолютные бенчмарки CPL расходятся в 3 раза по методологии — сравнение с рынком см. benchmarks.md |
| lead→qualified % | 2–4 недели, накоплено ≥15–20 лидов в ячейке | при 10 лидах один мусорный = 10 п.п. качелей |
| meetings / pipeline | месяц, когортно | qualification лагает 1–4 недели, deals — месяцы |

Правило: **алертим на первый сломавшийся слой воронки (§3), а не на CPL**. CPL — произведение слоёв; действие по нему напрямую лечит симптом, а не причину.

## 2. Операционный каденс

Пока MCP-аналитика (`campaign_perf`, `budget_pacing`, `conversion_funnel`, `analyze_lead_quality`, `leads_by_campaign`) — стабы (project-context.md), данные снимаем руками: доставка/спенд — Ads Manager UI или Graph API v25 insights по `act_776404808314031`; качество — CRM (общий Postgres, `v_leads_meta`). Первые 72 ч после запуска — отдельный режим, см. launch.md.

### Ежедневно (5 минут, только блокеры, НИКАКИХ правок)

- [ ] Delivery status всех активных адсетов = active? Застрял «In Review» >24 ч → эскалация через Account Quality (процедуры — policy.md).
- [ ] Spend pacing: тратится ≥90% дневного плана? Помни: Meta легитимно флексит **до +75% дневного бюджета** в сильные дни, недельный предохранитель — 7× daily (Вс–Сб). Однодневная недотрата — не сигнал; смотри неделю.
- [ ] Аномалии (кандидаты на автоматизацию в worker/Telegram по паттерну google-ads): CPM +30% день-к-дню; CPL >2× своего 7-дневного среднего; >50% дневного бюджета за первые 2 часа (порог калибруй с учётом легального 75%-флекса); нулевая доставка на активном адсете. Алерт — да, авто-kill — никогда.
- [ ] Лиды дошли до CRM? (webhook → `lead_resolve` → CRM; расхождение счётчиков = разбор сегодня).
- [ ] **Speed-to-lead**: все вчерашние лиды contacted? Латентность первого контакта — главный управляемый фактор качества и кормит ограничение «qualified в пределах 28 дней» (§5).

Вопрос дня один: «есть ли бинарный блокер?» Оптимизация ежедневно на наших бюджетах = сброс learning + шум.

### Дважды в неделю (Пн/Чт): бюджетные решения

Только в эти дни трогаем бюджеты. Scale/trim по 3–5-дневным окнам — правила и пороги в §4. Между ревью победителей не трогаем.

### Еженедельно (30–45 минут, глубокий разбор)

Открыть: Ads Manager (кампании по языкам + Breakdown by Placement + Delivery Insights на адсетах) и CRM-дашборд качества. Чеклист:

- [ ] 5-слойная декомпозиция по каждому адсету против его trailing-30d baseline: CPM → link CTR → form open rate → form completion → lead→qualified (CRM). Первый отклонившийся слой >25–30% — в дерево §3.
- [ ] Frequency (7d) ≤3.0–3.5? FTIR ≥25%? (Delivery Insights: Ad Sets → Ad Details; нужно ~5 дней доставки и 500+ показов.)
- [ ] Hook rate на видео (3-sec plays / impressions) ≥25%? Ниже — задача студии на новый опенер (creative.md).
- [ ] Placement breakdown: доля Audience Network = 0? (Gotcha про «5% на excluded placements» — §3.3.)
- [ ] CRM: lead→qualified % по рынкам/адам в коридоре; доля мусора; распределение speed-to-lead.
- [ ] Kill/refresh-решения по §4. **Все правки — одним батчем в этом окне**: сбрасывают learning свопы креативов, бюджетные скачки >~20%, смена оптимизации; добавление новых ads в живой адсет официально — significant edit, но на практике обычно даёт лишь мягкий/частичный сброс (directional) — всё равно только батчем в этом окне.
- [ ] Обновить weekly-отчёт (§6).

Вопросы недели: сломался аукцион (все рынки сразу) или наш креатив (один адсет)? Качество упало из-за placement, формы или ботов? Что из изменений прошлой недели дало эффект (single-variable — иначе не атрибутируешь)?

### Ежемесячно (стратегический)

- [ ] Reconciliation: CPL из Meta vs CPQL из CRM по каждому рынку; сверка числа лидов Meta vs строк CRM по `lead_id`-join.
- [ ] Когортная таблица: месяц лида × текущая стадия (§6) — единственный честный способ судить длинный цикл.
- [ ] EMQ-ревью (цель 7.5+ когда CAPI живой; для Instant Forms `lead_id` матчит почти детерминированно — §5).
- [ ] Attribution sanity: окно 7-day click, счёт First Conversion — не съехали?
- [ ] Learning Limited перепись: это норма или фрагментация сигнала? (§3.7).
- [ ] Концентрация креатива: 70% спенда за 90 дней в <5 ads = хрупкость аккаунта (directional).
- [ ] Пересборка бюджетов под сезонность → planning.md. Incrementality-проверки (geo-holdout по языковому рынку, вопрос «как вы о нас узнали» в скрипте квалификации) → дизайн и лог в learning.md.

## 3. Диагностическое дерево

База: **CPL = CPM / (1000 × link CTR × CVR)**, для instant forms — 5 слоёв: CPM → link CTR → form open rate → form completion rate → lead→qualified (CRM, шестой слой). Чинится первый слой, отклонившийся от собственного trailing-30d baseline. Никогда не действуй по CPL напрямую.

### 3.1 CPL взлетел

| Сигнатура слоёв | Диагноз | Действие |
|---|---|---|
| CPM +30%+, CTR/completion flat, все рынки сразу | Аукционная инфляция / сезон (Q4: CPM +40–66%) | Терпение, шире доставка или переброс бюджета. Креатив НЕ трогать |
| CPM ↑ + frequency >3.5 + FTIR <25% | Насыщение аудитории | Refresh креатива (§4); аудиторию не сужать |
| CTR −25%+ на креативе ≥3 недель | Fatigue — база пост-Andromeda (цикл 2–4 нед; на узких SR/HE — 1–3 нед) | Плановый refresh; творческая сторона — creative.md |
| Hook rate <25% и CTR низкий | Первые 2–3 секунды не работают | Новый опенер, не новый таргетинг |
| Hook rate высокий, CTR низкий | Провал середины видео / слабый CTA | Hold rate и retention curve; CTA-блок |
| Form open rate ↓ | Ад обещает не то, что форма | Сверить креатив ↔ оффер формы |
| Completion ↓ | Трение формы (каждое лишнее поле ≈ −10% completion) | У нас трение сознательное (Higher Intent) — сравнивай только со своим baseline |

Пост-Andromeda ловушка: слабый креатив сам поднимает CPM (качество креатива — вход в retrieval). Проверь hook rate ПРЕЖДЕ чем списывать всё на «аукцион подорожал».

### 3.2 Лидов нет вообще

Нет спенда через 1 ч — норма; через 24 ч — разбор в жёстком порядке (первые четыре — бинарные блокеры, видны за секунды):

1. **Статус ревью**: rejected / застрял In Review >24 ч → policy.md.
2. **Account spending limit** (Billing & Payment Settings) — молча стопит ВСЁ при живых бюджетах кампаний.
3. **Платёжный метод**.
4. **Расписание/дата старта** адсета.
5. **Cost-per-result cap ниже клиринга аукциона** → алгоритм просто не биддит. Снять cap, несколько дней Highest Volume, узнать реальный CPL, потом cap 1.2–1.5× наблюдаемого.
6. **Аудитория слишком узкая** (<100k; при наших country-level таргетингах редкость).
7. **Дубль в аукционе с собственным адом** (same-account dedupe).
8. **Регуляторные декларации**: без `dsa_beneficiary`/`dsa_payor` ады на EU не публикуются, API-сборка падает на валидации — наш случай, кампании собираются программно. Housing SAC — по двухзонной архитектуре policy.md: зона A (EU/UK) обязана нести `special_ad_categories=["HOUSING"]`, недекларированный housing-ад там ловит реджект; зона B (RS/ME/IL) — не ставить, иначе зря режем таргетинг (цена декларации ~+23% CPM, directional).

Если спенд ЕСТЬ, а лидов ноль — это kill-правило 2.5–3× CPL (§4): structural failure. Проверь саму форму (`leads test`, `follow_up_action_url`, приходит ли webhook → `lead_poll` сверка).

### 3.3 Лиды есть, но мусор

Причины в порядке частоты; сигнатуры читаются ДО звонка в CRM:

1. **Placement drift.** Breakdown → By delivery → Placement (или Insights API `breakdowns=publisher_platform,platform_position`): доля Audience Network / Reels overlays растёт, blended CPL при этом падает. **Gotcha 2026:** исключение плейсмента больше не исключает — дефолтная галка позволяет тратить до 5% бюджета на excluded placements в Leads-кампаниях. Жёсткий фикс — account-level блок в Advertising Settings → Placement Controls (перекрывает всё). Проверь, что AN фактически = 0.
2. **Pre-fill three-tap accidental leads.** Contact rate в CRM падает при растущем completion rate; 25–40% pre-fill лидов не узнают бренд при первом звонке (directional). Фикс: Higher Intent форма + 1–2 квалифицирующих вопроса (бюджет, срок) — фильтрует 30–50% мусора; наши формы уже `is_optimized_for_quality` + SMS-OTP (project-context.md).
3. **Бот-волна.** Сигнатура — burst: десятки почти одновременных лидов, ночные часы, повторяющиеся IP/паттерны, мусорные имена. Непропорционально приходит из AN. Дёшево ловится SQL-детектором в CRM: >N лидов с одного ада за 10 минут → алерт.
4. **Один рынок портит blended** — `breakdowns=country`.

Ограничение API: у лида из leadgen Graph API есть ad/adset/campaign/form_id, но **нет placement** — качество-по-плейсменту только через агрегатный breakdown против CRM qualified-rate по ad, либо временная изоляция подозрительного плейсмента в отдельный адсет.

Структурный фикс хронического мусора один — Conversion Leads + CAPI-фидбек (§5): единственный механизм, которым оптимизатор вообще узнаёт о качестве.

### 3.4 Расход не тратится / недотрачивается (<90% бюджета)

- Прогнать пункты 1–8 из §3.2 (чаще всего: cap, аудитория, DSA).
- Aspirational cap — главный виновник на малых бюджетах: cap ниже рынка = стоп доставки и вечный learning. Якорь — trailing-30d actual +10–20%, не хотелка.
- Не путать с флексом: Meta законно тратит до +75% в день и компенсирует в рамках недели — суди pacing по неделе, не по дню.

### 3.5 Частота растёт, CTR падает

- «Ножницы» (CPA ↑ при frequency ↑) = подтверждённое насыщение/fatigue.
- Проверка: FTIR <20–25% (ранний флаг — <50%), audience reached >60–70% в Delivery Insights. Один аудит 2025: адсеты с FTIR <15% показывают CPA минимум +35% против FTIR >40% (directional).
- Действие: refresh новыми КОНЦЕПТАМИ (Andromeda кластеризует near-duplicates — вариации не считаются). Аудиторию не сужать. Малые рынки (SR, HE) выгорают быстрее всех — refresh планируй заранее, а не по факту распада.

### 3.6 Auction overlap

Пост-Andromeda аукцион мягче дедуплицирует свои же ады — overlap проверяем ПОСЛЕ насыщения (FTIR/frequency), не первым. Overlap >20% в Delivery Insights + недотрата/рост CPM → консолидировать адсеты. У нас 1 адсет/язык, но пересечение возможно между языковыми кампаниями на общих geo (RU-диаспора и EN в одной стране) — структурный ответ в planning.md.

### 3.7 Learning Limited

- Порог не изменился: 50 optimization events / 7 дней на адсет. Выход требует дневной бюджет ≈ (CPL × 50) / 7 — на наших бюджетах арифметически невозможен. **Хронический Learning Limited — наш штатный режим**; прибыльные аккаунты годами так живут. Встречается и мягче: «10 конверсий за 3 дня» у части аккаунтов, «15–25 events/нед часто хватает» (directional).
- Решение: CPL в таргете → игнорируй лейбл. CPL плохой И 3+ адсета собирают <20 events/нед на пересекающихся аудиториях → консолидировать (3×20 фрагментов не выйдут из learning никогда; 1×60 выйдет). Событий мало потому, что оптимизация слишком глубокая → подними event на уровень выше (Leads вместо qualified) до появления объёма.
- НЕ реструктурировать ради лейбла «Active» — это лейбл плотности данных, не KPI.

### 3.8 После изменения всё сломалось

1. Что именно меняли? Learning сбрасывают: своп креатива, бюджет >~20%, смена оптимизации/события. После сброса — 3–7 дней шума: не трогать, не наслаивать правки.
2. Правок было несколько сразу → эффект не атрибутируется. Впредь: батч раз в неделю, single-variable.
3. Никаких выводов до дня 5; если после 7–14 дней стабильно хуже baseline → откат к последней рабочей конфигурации (лог изменений — learning.md).
4. Сломалось после включения Conversion Leads: рост raw CPL — **ожидаемое поведение**, суди по CPQL после 2+ недель стабилизации (§5).

## 4. Kill / Scale / Refresh

### Kill

| Правило | Порог | Механика |
|---|---|---|
| Structural kill (в любой момент) | Потрачено **2.5–3× target CPL, лидов 0** | Это сломанная форма/воронка/DSA, а не «мало данных». Pause + разбор по §3.2 |
| Окно невмешательства | Дни 1–2 — шум; **никаких решений до дня 5**; первое настоящее окно оценки — дни 7–14 | Однодневные качели — главный источник преждевременных kill |
| Observe | CPL +10–30% над таргетом | Наблюдать ещё 3 дня, не трогать |
| Performance kill | CPL **>50% над таргетом после ~дня 10** на 3–5-дневном окне | Pause. Никогда — по одному дню |

Автоматизация (Marketing API + Telegram-лог): у каждого automated-правила обязательное условие «адсет не в learning phase» (кроме structural kill); pacing-аномалии из §2 — только алерт, авто-kill по ним запрещён. Каждое авто-действие — в лог на ревью оператора.

### Scale

- **Только Пн/Чт.** Шаг **+10–20% раз в 48–72 ч**, только на адсетах со стабильным 3-дневным CPL на уровне или ниже таргета.
- Почему ≤20%: скачки >20–30% сбрасывают learning и поднимают CPA на 25–40%. С 2025 потолок мягче — Ads Manager показывает account-specific «safe increase to X»: если показан, используй его число. Железным осталось не число, а **каденс**: фиксированные дни ревью, между ними победителей не трогать.
- **Дублировать vs поднимать бюджет:** пост-Andromeda дублирование адсетов/кампаний фрагментирует signal density — почти всегда поднимай бюджет вертикально в существующем адсете. Горизонтальный рост = новые креативные концепты и следующий языковой рынок (planning.md), не клоны победителя.
- Bidding: дефолт — **Highest Volume**. Cost-per-result goal — только после 30+ дней стабильного CPL, cap на **10–20% ВЫШЕ** trailing-30d actual. Cap «по хотелке» = стоп доставки (§3.4).

### Refresh

- Ожидаемый цикл жизни креатива пост-Andromeda: **2–4 недели**, на узких аудиториях (SR, HE) — 1–3; пик обычно в первую неделю, «не выстрелил за неделю — вряд ли выстрелит» (directional).
- Триггеры — сигнальные, не календарные: **frequency >3–4; CPM +15% sustained; CTR −20%; FTIR <25%**. Но производство — на опережение: конвейер студии должен выпускать концепты ДО распада текущих (каденс и креативная сторона — creative.md).
- Механика: **добавляй новые ads в живой адсет** — официально это significant edit, но с 2025 на практике обычно не сбрасывает learning целиком (directional) и сохраняет накопленный сигнал; добавляй батчем в недельное окно, параллельные тест-структуры не плодить.
- Держи 5–10 живых, действительно РАЗНЫХ концептов на язык (отмасштабировано под наши бюджеты от практики «6–10 prospecting-концептов»); вариации-косметика кластеризуются Andromeda и дополнительной доставки не покупают.

## 5. Сигнальная гигиена

### CAPI for CRM — обязательный путь

Offline Conversions API мёртв с мая 2025; все CRM-события идут через стандартный Conversions API: одно событие на переход стадии, `action_source='system_generated'`, `event_time` = момент перехода, `user_data` = **`lead_id`** (15–16 цифр, деterministic-матчер) + SHA-256 email/phone. Наша таксономия стадий, суммы и инвариант «событие уходит только при наличии `meta_lead_id`, никогда не матчить по хешу PII» — project-context.md; транспорт — воркер `capi_drain`. Минимум — раз в сутки; у нас near-real-time по смене lifecycle — лучше и сразу закрывает требование Conversion Leads о ежедневности.

- В Events Manager собери стадии в funnel и пометь qualifying vs disqualifying. Верификация — до ~1 дня на каждый event type: закладывай при добавлении стадий.
- Ресёрч: негативная стадия (`lead_disqualified`) как non-qualifying заметно помогает модели отсеивать мусор. В текущей таксономии её нет (SKIP-список — project-context.md) — кандидат на добавление; решение и результат зафиксировать в learning.md.

### EMQ

Шкала 0–10 на событие в Events Manager; цель **6+, лучше 8+** (в месячном ревью — 7.5+). Приоритет матчеров: hashed email > hashed phone > external_id > fbc/fbp > IP+UA; стек из нескольких бьёт любой одиночный. Для наших Instant-Form-лидов `lead_id` матчит почти идеально — EMQ-тревога актуальна, только если появится web-form путь (локализованные лендинги): тогда с первого дня захват `fbclid`/fbp на сабмите и общий `event_id` pixel+server.

### Дедупликация

- Meta склеивает событие при **идентичных `event_name` + `event_id`** от pixel и сервера; ломается на самогенерённых ID, лишних символах и регистре (`lead` ≠ `Lead`).
- Наши CRM-стадии — server-only (`system_generated`), браузерного близнеца нет → **дедуп им не нужен**.
- Железное правило: **никогда не вешать браузерный Lead-pixel на лиды, приходящие через leadgen-webhook** — двойной счёт лидов в отчётности.

### Атрибуция для длинного цикла

- С 12 янв 2026 максимум по view — 1 день (7d/28d-view удалены); с 3 мар 2026 клик = только реальный link click (лайки/шеры не считаются). Исторические сравнения через эти границы сломаны.
- Настройка: **7-day click, без view** — обдуманная покупка за €263k+ не view-driven, view-through = шум. Счёт — **First Conversion**.
- Прими недоучёт как данность: за 7-дневным окном Meta слепа, 20–40% конверсий длинного цикла не кредитуются. Всё дальше 7 дней живёт в CRM по `lead_id`. 28-day-click частично доступен через Marketing API для сверок, хотя UI его прячет.

### Conversion Leads optimization — требования и момент переключения

| Требование | Порог |
|---|---|
| Объём лидов | ~200–250/мес на уровне dataset (не адсета) |
| Оптимизируемая стадия | наступает **в пределах 28 дней** от создания лида |
| Конверсия в стадию | 1–40% |
| Отправка событий | минимум ежедневно, стабильно |
| `lead_id` | 15–16 цифр, сохранён в CRM |
| Стадий шлётся | ≥2; формы — только native Instant Forms |

Плюс ориентир ~50 optimized-stage events/нед на адсет для выхода из Learning Limited (не gate, но влияет на доставку).

Наша цель оптимизации — **`lead_qualified`**: при SLA первого контакта 3–7 дней стадия укладывается в 28-дневное окно. Meeting/deposit/contract шлём для отчётности и обучения воронки, но оптимизироваться на них нельзя — они за окном.

Последовательность переключения с raw Leads:

1. `lead_id` пишется на каждый лид, события стадий верифицированы в Events Manager.
2. 2–4 недели копим историю стадий, оставаясь на Leads goal — Meta нужна воронка.
3. Флип **только highest-volume адсетов** (по объёму qualified — вероятно RU/EN); 2+ недели стабилизации до любых суждений и скейла.
4. Ожидаемый результат: raw CPL **растёт**, CPQL падает на 15–21%. Предупреди стейкхолдеров ЗАРАНЕЕ — рост CPL не регресс, а смена цели.

Рынок не даёт ~1 qualified/день → остаётся на Leads goal + трение формы (Higher Intent, квалифицирующие вопросы) как интерим-фильтр качества.

Пост-Andromeda контекст: сигнал кормит retrieval — чистый ежедневный CAPI ускоряет обучение и удешевляет доставку. Сигнальная гигиена — рычаг доставки, а не бухгалтерия.

## 6. Отчётность

### Минимальный дашборд (строка = язык-рынок)

| Поле | Источник |
|---|---|
| Spend | Insights API / Ads Manager |
| Leads (**First Conversion**) | Insights API |
| CPL | spend / leads |
| Qualified leads | CRM (`v_leads_meta` × lifecycle) |
| **CPQL** | spend / qualified — главная рулевая метрика |
| Lead→qualified % | CRM |
| Meetings | CRM |
| Cost per meeting | spend / meetings |
| Speed-to-lead (медиана) | CRM |

Итоговая строка: blended по аккаунту + **pipeline value MTD** по рынкам. Один и тот же дашборд смотрят маркетинг и сейлзы — качество лида общая ответственность, и qualified-разметка в CRM должна быть живой, иначе CPQL — фикция.

### Формат «наши цифры vs план vs рынок»

На каждую метрику три колонки: **actual / план / рынок**. План — из медиаплана (planning.md). Рынок — из benchmarks.md, с обязательной оговоркой: real-estate CPL-бенчмарки расходятся до 3× из-за определения «лида», поэтому первичное сравнение — со **своим trailing 4-week baseline**, рынок — только рамка. Отклонение от плана >25–30% сопровождается диагнозом по дереву §3 (какой слой сломался), а не голой цифрой. Весь отчёт — одна страница.

### Еженедельный отчёт

1. Таблица по рынкам (выше) + WoW-дельты.
2. Topline: spend / leads / qualified / CPQL vs план.
3. Действия недели: kill / scale / refresh — одним батчем, с причиной по §3–§4.
4. Диагнозы аномалий (первый сломавшийся слой, не симптом).
5. Риски и очередь креативов от студии (fatigue-прогноз по frequency/FTIR).

### Ежемесячно добавляется

- **Когортная таблица: месяц лида × текущая стадия.** Qualification лагает 1–4 недели, сделки — месяцы; судить кампанию по same-week данным о сделках статистически бессмысленно — только когорты.
- Reconciliation Meta vs CRM (лиды по `lead_id`-join, CPL vs CPQL по рынкам).
- Pipeline value по рынкам — north star: kill/scale только по CPL систематически перекачивает бюджет в рынки самых дешёвых лидов, а не самого богатого пайплайна.

Сборка сейчас: insights — Graph API v25 (First Conversion) + SQL по общему Postgres; когда MCP-аналитика оживёт (`campaign_perf`, `analyze_lead_quality`, `leads_by_campaign`) — тот же дашборд собирается ею без смены структуры.
