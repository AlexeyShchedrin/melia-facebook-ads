# Meta (Facebook) Ads — автоматизация: MCP + Marketing API. Исследование

> Собрано: **2026-07-01**. Актуальная версия API на момент сбора: **Graph / Marketing API v25.0** (вышла 2026-02-18; v26.0 ожидается ~сентябрь 2026).
> Источник данных: многоагентное исследование (7 направлений + блок верификации против официальных доков Meta и реальных репозиториев).

## Цель проекта

Построить инструмент, который с локального компьютера умеет:
1. Загружать креативы (картинки/видео) в Meta.
2. Создавать кампании с нужными таргетингами.
3. Управлять кампаниями (пауза/запуск/бюджеты/дубли).
4. Следить за результатами (метрики/инсайты).
5. Подключать Lead-формы.
6. Собирать лидов в **самописную CRM**.

## Главные выводы (TL;DR)

- **Единого готового «всё-в-одном» нет.** Фундамент под все задачи — **Meta Marketing API** (часть Graph API). Он покрывает 100% требований.
- **MCP-серверы — надстройка-удобство** поверх того же API (управление на естественном языке). Есть **официальный** MCP от Meta. Но:
  - ни один MCP не закрывает весь пайплайн;
  - самое слабое место у всех MCP — **Lead-формы и сбор лидов** (официальный Meta MCP их **не делает вообще**).
- **Лиды в свою CRM** собираются НЕ через MCP, а напрямую: вебхук `leadgen` → `GET /{leadgen_id}?fields=field_data` → запись в БД. Это надёжнее строить самому.
- **Главный барьер — не код, а доступы:** App Review + Business Verification. **Лазейка:** для СВОИХ рекламных аккаунтов в СВОЁМ Business Manager всё работает **без App Review** (System User token + расшаренные ассеты).
- **API бесплатный.** Платного тарифа нет — платишь только за показы рекламы. Более высокие рейт-лимиты даются бесплатно за налёт вызовов.
- **«Локально» вебхуки принять нельзя** — нужен публичный HTTPS-эндпоинт (ngrok / Cloudflare Tunnel / VPS). Либо только polling.

## Рекомендуемая архитектура

```
[Локальный CLI/скрипт: Python + facebook-business SDK]
   ├─ upload креативов (adimages / advideos) ────┐
   ├─ create Campaign → AdSet(targeting) → Ad ────┤→ Marketing API (System User token)
   ├─ manage (pause / copy / budget) ─────────────┤
   └─ insights (async reports) ───────────────────┘

[Небольшой ПУБЛИЧНЫЙ сервис: VPS / Cloud Run / Cloudflare]
   ├─ POST /webhook  ← leadgen (проверка X-Hub-Signature-256)
   │     → GET /{leadgen_id}?fields=field_data
   │     → upsert в CRM (дедап по leadgen_id)
   ├─ cron: poll GET /{form_id}/leads (добор пропущенного, окно < 90 дней)
   └─ (позже) Conversions API for CRM — события качества лида обратно в Meta
```

## Дорожная карта (порядок запуска)

1. **Свой Business Manager** → создать app типа Business → добавить Marketing API.
2. Сгенерировать **System User token**, расшарить свой ad account + Page.
3. Прогнать полный цикл на СВОЁМ аккаунте (без App Review): загрузка креатива → кампания `OUTCOME_LEADS` → объявление в `PAUSED`.
4. Настроить приём лидов: подписка на `leadgen`, проверка на **Lead Ads Testing Tool** (без трат бюджета).
5. Добавить **Conversions API for CRM** для оптимизации на качество лидов.
6. App Review + Business Verification — **только** когда пойдёшь на клиентские аккаунты.

## Ключевые предостережения (из верификации)

- Официальный Meta MCP **НЕ поддерживает** Lead-формы/выгрузку лидов (подтверждено).
- Подпись вебхука — **`X-Hub-Signature-256`** (HMAC-SHA256 по сырому телу), не устаревший `X-Hub-Signature`/SHA1.
- Тиры доступа Meta переименованы (04.05.2026): `Standard Access` → **`Limited Access`**, `Advanced Access` → **`Full Access`** (маппинг 1:1).
- Порог для повышенного рейт-тира снижен: **500** вызовов за 15 дней (было 1500), ошибок < 15%.
- Точные строки разрешений: `leads_retrieval` (не `lead_retrieval`), `ads_management`, `ads_read`, `business_management`, `pages_show_list`, `pages_read_engagement`, `pages_manage_metadata`.
- Ретеншн лидов **90 дней** — забирать сразу.
